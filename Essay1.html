Implementation of ASMIS into Queens Medical Centre

 

Overview

 

            Queens Medical Centre (Queens) is a clinic that serves the local community.

The current procedure when a patient needs an appointment with a specialist, requires them to phone Queens and speak directly to the receptionist. This process is not efficient due to the high number of calls currently received by Queens, and some patients have experienced delays in getting access to specialists. With the expected growth of the local community, a better solution needs to be defined.

The proposal put forward by Queens to modernize their appointment system, is the implementation of a web-based Appointment and Scheduling Management Information System (ASMIS).

 

 

Benefits and Risks

 

An ASMIS, will streamline the current process, and make it easier for the patients to get treatment.

·      Patients will be able to book appointments, without having to speak to the Receptionist.

·      Patients will include their symptoms when booking, and this will aid in selecting a suitable specialist.

·      Using ASMIS, the patient can also change their appointment.

·      The “human factor” is removed, and appointments can be booked 24/7.

 

Big areas of concern with any online activity are Data Protection and Cyber Crime.

In this case, both factors are high profile, as the data is personal health and medical data. Any leak of this information, would be damaging for Queens and their reputation, and they would lose the trust of the local community. There are also likely to be legal implications associated with any data breaches of confidential data.

           

            Though not stated directly, it is likely that the database used for the ASMIS, will also contain detailed information on the patient’s medical history. This would be logical, as the person making the referral to a specialist, would need to have all possible information at hand, in order to make a more effective referral.

This makes sense from an ASMIS perspective, but increases the security risk, as if the database is compromised in any way, the attacker could potentially have access to the complete medical history of the Patient/s contained within.

 

 

With healthcare data breaches becoming increasingly common (FierceHealthcare 2022), it is vital to implement a secure and robust solution.

With this in mind, this report will detail possible solutions on how to implement an ASMIS, and how to keep it secure against all threats.

 

 

Solution Outline

It is a high-level checklist, to ensure that what is deployed, has taken account of the different potential threats, and has included mitigation against them.

The STRIDE model will be used as a guide for the secure deployment of ASMIS.

As the ASMIS Database would be “Internet facing”, this would subject it to constant random cyber-attacks, in addition to potential targeted attacks, and thus highlights the necessity for a robust security solution.

 

            The following sections outline the strategy for the secure implementation of ASMIS into Queens.

 

User registration and Authentication

            An important factor is the patient’s usage of ASMIS.

Whilst designing a solution, we have to be mindful of the practicalities of some of the technology, as well as the likely diverse user base, with varying IT skill levels, that need to be able to use ASMIS.

            Initial registration for ASMIS should be performed using the mail system or “snail mail”. This would consist of a username and initial password combination, being sent to the registered home address of the patient. Whilst not infallible, it mitigates the chances of an attacker being able to impersonate the patient, in order to obtain their personal login credentials. This registration letter, would only provide initial access to the ASMIS portal, where the user would then be expected to change their password, to a new one with a minimum level of complexity.

            Multi-Factor Authentication (MFA) known also as 2 -Factor Authentication (2FA) should at this point be implemented (Nist, 2022). A good option is to register a personal Cell Phone number in the system. When the user logs in, the ASMIS authentication process, would send a PIN code to the phone, that is only valid for a short time. The patient would then use this PIN in conjunction with their username/password, to authenticate more securely. Additionally with MFA, the use of “personal questions” which would be presented to the user on initial login are also required. There would be a pool of questions that require personal responses relevant to the patient, and on login, 3 random questions from the pool, would need to be answered successfully before authentication is confirmed.

            The specifics of MFA implementation, need to be carefully evaluated, especially with regards to device incompatibility, preventing successful logins, as this would be detrimental to the overall effectiveness. Personal equipment, such as PCs or Tablets, that do not meet a “Corporate standard” would be used with ASMIS, and incompatibility could cause issues.

            Whilst MFA is still not 100% foolproof, and it can often be seen as a nuisance by end users (ITproportal, 2019), it adds another layer of security to the initial login process. A potential attacker would need to know personal information and/or have in their possession, the cell phone (or a cloned SIM) of the patient, in order to gain access. Personal information could also be obtained through phishing, but the use of MFA is still preferable to not.

            Biometric authentication or hardware keys that generate a random code, would also be possible, but the overhead and costs of implementing this are prohibitive.

            Due to some negativity around MFA usage, as part of the “signup” process, it should be explained to the patients, how this will benefit them and safeguard their data (Covrsecurity, 2020).

 

 

 

Database Protection

            The database needs to be secure (VPNmentor, 2022), for patient confidentiality, data protection and for legal reasons, such as the EU wide General Data Protection Regulation (GDPR) which regulates the use of data (gdpr.info, ND).

            The ASMIS database is designed to be used by patients that are remotely located from Queens, and means it will be hosted and presented directly to the public internet, using a public IP address. This increases the risk of cyber-attacks, as it is contactable by anyone with an internet connected device.

            There are a number of approaches, that can help to limit unauthorized access to the database.

            The database should be contactable only via a secure encrypted connection. This is achieved using HTTPS and SSL, which encrypts the communication.

            Non-default TCP ports should be used for connection (Sonicwall, 2019). This is an element of “security through obscurity”, but a lot of attacks are launched on common TCP ports, waiting for an expected HTTPS, DNS, FTP etc. response. If the TP port used is changed, it will prevent some attacks just by this simple step.

            The database itself should be encrypted. 

            The web server and the database should be on different servers. Doing this will enable the Web Server to accept the Patient requests, but the database server, can be more secure behind a Firewall.

            Access to the database needs to be highly restricted, allowing access to the areas that are required by that particular user. For patients, this means that they would only be able to see their records, but there also needs to be tight control internally, to ensure that only authorized personnel have access to ASMIS. A minimum number of Administrators should have admin rights to the database for support.

            Both the ASMIS database and the fileserver hosting it, need to be continually patched and updated as part of a regular security patching schedule. This is required in order to prevent vulnerabilities, and/or for general improvements to the functionality via the updates.

            The database should be periodically “backed up” to a storage medium, that can be stored offsite at another location. This ensures that if the database is corrupted or affected by a successful attack, it can be recovered and the data restored.

            Database access requests, both success and failure should be continually logged and reviewed. This helps identify unusual patterns of access requests.

 

 

 

 

 

 

 

 

 

 

 

 

 

Network Infrastructure Protection

            The Network Infrastructure must be protected using a multi-point strategy.

            A firewall should be implemented as close as possible to the entry point from the Public Internet. The Firewall helps prevent the proliferation of Viruses, installation of Malware and Spyware, and can prevent many forms of hacking attempts (Fortinet, ND). Installation of a Network Firewall should be considered mandatory.

            Strong consideration should be given to implementing an IDS/IPS device into the Network. IDS/IPS has the ability to dynamically learn and identify suspicious Network activity, and in the case of IPS, block it without further action from the IT Team. IPS and Firewalls should be considered as a consolidated approach to Security, as their functions are essentially intertwined with each other’s. Additional care should be taken when implementing IPS as it requires adequate bandwidth to avoid resource overflow with the Network performance suffering, and it also can deny legitimate traffic (Rapid 7, 2020).

            A further device for implementation is a Web Application Firewall (WAF). This differs from a traditional Firewall (Cloudfare, ND), in that its function is at Level 7 of the OSI model, and will directly protect the traffic destined for ASMIS itself, by analyzing the HTTPS communication. A traditional Firewall is unable to provide this protection. Protecting web access to the ASMIS database should be considered crucial.

            It is recommended to install a Next Generation Firewall (NGFW) at Queens. A NGFW (digitalguardian, 2020) is a device that combines all of the features of a traditional Firewall, IPS and WAF, into a single appliance. A single appliance with all of these features implemented, would help to secure fully the ASMIS database, and will help to streamline the required support from the IT Team.

The logging features of all of these devices, can help detect, identify and safeguard against unauthorized access.

 

Testing and verification methodologies

            Once the ASMIS database has been deployed, consideration should be given to employing White Hat Hackers (Techjury, 2022) to test and verify that the installation is secure. White Hat Hackers will try and gain unauthorized access to both ASMIS and the hosting infrastructure, using penetration testing methodologies, and they will identify any weaknesses.
